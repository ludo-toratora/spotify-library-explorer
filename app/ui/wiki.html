LibraryExplorer

Upload your Spotify library once. Then navigate your music through similarity graphs, embeddings, and genre clusters. Build playlists by selecting communities.

Why I Built This

Once my Spotify library passed 7,000 liked songs, maintaining playlists manually stopped being practical. I looked for tools that could help explore a library that size, but most recommendation systems were built around popularity, and the few analysis tools I found did not work well for larger collections.

I started with a genre graph to understand the structure of my own listening habits. It helped, but some areas were too dense to read: cool jazz, bebop, big band, hard bop, free jazz, jazz fusion, and jazz funk all ended up crowded into the same region.

To keep exploring, I added artist similarity graphs and UMAP embeddings to break those dense areas apart and inspect the library at a finer level. What started as a script for my own use eventually made more sense as an app that others could use too.

Getting Started

Before you can explore, you need to upload your library and run the analysis pipeline. This only happens once. After that, all views load instantly from cache.

1
Export your Spotify liked songs

Go to chosic.com/spotify-liked-songs, connect your account, and download as CSV. This gives you all your liked tracks with audio features and genres.

2
Upload in Settings

Go to Settings → Upload. Tick Chosic export (.csv), then drag your file into the drop zone. LibraryExplorer validates it and shows any warnings before you proceed. If you already have a normalized_tracks.json from a previous run, upload that directly without the toggle.

3
Run the pipeline

Click Compute. The pipeline builds similarity graphs, detects communities, and computes UMAP embeddings. For a library of ~2,500 artists this takes 10–45 minutes. You can watch progress live at Pipeline Progress.

4
Start exploring

Once the pipeline finishes, the Start Exploring button appears on the home page. From there, choose any of the three explore views.

Exploring Your Library

There are three views, each showing your library from a different angle. All three share the same selection model: anything you select ends up in Selected.

Selecting artists

In any Explore view, you can build a selection several ways:

  • Click a node to select it. Shift+click to add more without losing the current selection.
  • Drag a box over an area to select everything inside it.
  • Click Add Connected in the bottom bar to pull in all graph neighbours of your current selection.
  • Click Add Visible to grab everything currently shown on screen (respects filters).

Once you have a selection, click Add to Selected in the bottom bar. It lands in Selected as a named set. The name is auto-generated from where it came from.

Artist Graph

A force-directed network where nodes are artists and edges connect similar ones. Artists with strong mutual similarity pull together and form colour-coded communities. The layout settles via physics simulation. Drag nodes, scroll to zoom, and drag the background to pan.

The left panel lets you:

  • Switch between the 5 similarity presets (Balanced, Sound Alike, Genre Match, Same Era, Vibe & Era). Each produces a different community structure
  • Change colour mode to highlight by community, parent genre, decade, energy, or bridge artists
  • Toggle physics on/off or hit Lock Positions to freeze the layout
  • Show/hide communities from the cluster list, useful for focusing on a subset

The right panel shows info about the hovered artist and stats about the current selection. Bridge artists (those connecting multiple communities) are highlighted when you switch to the bridger colour mode.

Genre Graph

Genre labels are lossy: Spotify assigns them inconsistently, and the same string covers acoustically unrelated artists. This view shows co-occurrence structure instead: what actually sits next to what in your library. Nodes are genres, sized by how many tracks you have in that genre. Edges connect genres that frequently co-occur on the same artist. The heavier the edge, the more often they appear together. Louvain communities group genres into families.

Use this view to understand the shape of your taste: which sub-genres sit at the centre of your library, which are isolated niches, and which act as bridges between very different musical worlds. The left panel controls are the same as the Artist Graph.

Embedding Map

A 2D scatter plot where every dot is an artist. Position is determined by UMAP, a dimensionality reduction that tries to place similar artists close together. Unlike the graph, there are no explicit edges: proximity itself is the signal.

The right panel shows axis correlations: which audio features (energy, tempo, valence, etc.) most strongly drive the X and Y axes of the current embedding. Switch between the Audio Only and Combined embedding presets in the left panel to see how the map changes when genre and era information is added.

Colour modes work the same as in the graph views. Box-select a cluster of dots to add that region to Selected.

Building a Playlist

The Selected page is where you assemble, refine, and export your playlist. Everything you add from Explore or Browse lands here as a named set. Sets combine into the final result: union of all active sets, duplicates removed.

Working with sets

Each set card shows where the artists came from, how many there are, and what filters (if any) are active. You can rename a set by clicking its title, duplicate it, or delete it. The combined result in the right panel is the union of all active sets, with duplicates automatically removed.

Filtering a set

Click the filter icon on any set to open the filter editor. You can restrict by audio features (energy range, tempo range, valence, danceability) or metadata (decade, minimum popularity). Filters show a live preview of how many artists remain. Click Reset Filters on a set to restore all original artists.

Reviewing artists

Click Review Artists to open a track list grouped by artist. Uncheck any artist to exclude them from the export. They stay in the set but are marked as excluded. Useful for removing a handful of artists that don't fit without re-doing the whole selection.

Metrics

The right panel updates live as you edit sets. It shows total artist and track count, diversity metrics (genre spread, decade range, energy/tempo variance), dominant characteristics, and how many communities and parent genres are represented.

Exporting

Click Export to choose a format. Spotify playlist creates a playlist directly in your account via the Spotify API. You need to connect your account in Settings first (see below). CSV (clean) exports a TuneMyMusic-compatible file you can import to any service. CSV (debug) includes audio features per track if you want to inspect the data.

You can also add artists to Selected from the Browse page. Use the Add to Selected buttons next to communities, genres, decades, and audio ranges without going through an Explore view at all.

Similarity weights

Each graph preset has sliders for audio, genre, and era weight. Drag them to adjust what drives similarity for that preset. Changes don't take effect until you hit Save & Recompute. Only the affected presets are recomputed, not the full pipeline.

UMAP parameters

n_neighbors controls how much local vs. global structure is preserved (15 is a sensible default). min_dist controls how tightly dots cluster together. Lower values (0.05–0.1) produce tighter clumps, higher values (0.3–0.5) spread things out.

Under the Hood

Genre as signal, not category

Genre was the starting point. The genre graph maps co-occurrence: which labels appear together on the same artists, how dense those connections are, which genres are isolated niches and which act as bridges. That structure is real.

The problem is using genre labels as the clustering criterion directly. "Jazz" aggregates free improvisation, smooth jazz, jazz-funk, and acid jazz. Acoustically unrelated artists, same tag. Clustering on genre labels produces groups that are meaningful as marketing categories and meaningless as audio collections.

The artist graph and UMAP use genre overlap as one of three similarity signals alongside audio features and era, not as the grouping criterion itself. Community structure emerges from acoustic proximity instead. The two views answer different questions: the genre graph shows how the library is labelled; the artist graph and embedding show what it actually sounds like.

From tracks to artists

Tracks are grouped by artist_name. Eight audio features are averaged across all of that artist's tracks: energy, danceability, valence, acousticness, instrumentalness, speechiness, liveness, and tempo. Genre tags are merged via set union across all tracks, then deduplicated. The primary decade is the mode decade across track release years; mean_year is the arithmetic mean.

Tempo is the only feature requiring normalization before aggregation: raw BPM (typically 40–200) would dominate cosine similarity over the other seven features, which are already in [0, 1]. It is capped at 200 and divided by 200.0 before being stored in the audio profile.

Similarity

Artist similarity is a weighted combination of three independently computed scores. Audio: cosine similarity over the normalised 8D feature vector. Genre: Jaccard overlap on tag sets, |A ∩ B| / |A ∪ B|. Era: linear decay capped at zero, max(0, 1 − |year_a − year_b| / 60), so artists 60 or more years apart score 0 regardless of everything else. These three are deliberately orthogonal: two artists can score high on audio and zero on genre, which is why different preset weightings produce structurally different communities rather than just visual variation.

Weights are normalized to sum to 1.0 before applying, so a preset like (0.7, 0.2, 0.1) is equivalent to specifying proportions. The composite score is a simple dot product of the three component scores with the normalized weights.

Graph and communities

A k-nearest-neighbours graph is built per preset (k=15, minimum edge weight 0.3). Edges below 0.3 are noise at this library scale: they connect artists with no meaningful acoustic or genre overlap and add graph density without community signal. Each edge stores the combined weight plus the three component scores, so the UI can display what is actually driving a connection. Edges are deduplicated on the sorted artist pair, so the graph is undirected.

Community detection is a custom Louvain implementation: greedy single-node reassignment, iterating until no further modularity gain is possible or a maximum of 10 passes. No resolution parameter. The modularity gain formula is the standard Louvain approximation: gain = edge_weight − (strength(node) × strength(community)) / (2 × total_weight). Communities are auto-named from their dominant parent genre, decade, and sub-genre tags.

Four centrality metrics are computed per node via NetworkX: betweenness centrality, PageRank, eigenvector centrality, and clustering coefficient. Bridge artists shown in the UI are those with high betweenness relative to their community; they sit on shortest paths between communities rather than at a community's centre.

UMAP

UMAP projects artist feature vectors into 2D while preserving local structure. Default params: n_neighbors=15, min_dist=0.1, metric=cosine. Output is centered and normalized to [−1, 1] on both axes.

Audio-only embeddings use a 6D vector: energy, danceability, valence, acousticness, instrumentalness, tempo. Speechiness and liveness are excluded: both are low-variance across most music libraries and degrade cluster separation when included.

Genre embeddings use a one-hot vector over parent genre labels (not raw Spotify tags). Each artist's vector is row-normalized to sum to 1.0, which removes the confound of artists having more tags overall. Era embeddings use a 7-bucket one-hot over decades (1960s–2020s).

Combined presets concatenate all three after per-component scaling. Audio and era components are multiplied by sqrt(n_genre_dims / n_component_dims) before concatenation. Without this correction, the genre one-hot (which can have many more dimensions than audio's 6) numerically dominates the combined vector regardless of the weight parameter; the per-dimension influence is equalized rather than the per-component influence.

The choice of input features matters substantially. Audio Only produces a near-continuous blob with no stable cluster boundaries (silhouette undefined at any reasonable k). Combined (audio + genre + era, balanced weights) resolves 18 clusters at silhouette 0.51. Genre-only reaches 0.54 with finer granularity but collapses the acoustic variation that makes playlists actually sound coherent.

Clusters on the 2D UMAP output are computed separately using DBSCAN (eps sweep over [0.08, 0.12, 0.18, 0.25], accepting the first result with at least 3 clusters), with k-means as a fallback if DBSCAN produces fewer. These 2D clusters and the Louvain graph communities have an adjusted Rand index of 0.16: broad agreement, boundary disagreement. They optimise different objectives, which is why both views exist.

Cache model

Each pipeline step caches independently. The cache key is the first 8 characters of SHA256(input file) joined with the first 8 characters of SHA256(relevant config section), stored as a 17-character string in a per-step manifest. If both hashes match the manifest, the step is skipped entirely.

This means changing similarity weights invalidates the graph and embedding caches but not artist aggregation (which depends only on the input file). Changing the input file invalidates everything. The full pipeline order is: load → validate tracks → aggregate → validate artists → build graphs (per preset) → detect communities → compute centrality → compute embeddings (per preset) → genre graph.

FAQ

Does it sync with Spotify automatically?

No. You upload your data manually and trigger the pipeline when you want to refresh. The Spotify OAuth is only used at export time to create playlists.

Why does the graph look different every time I reload?

The force-directed physics simulation is non-deterministic. Once you're happy with the layout, hit Lock Positions in the left panel to freeze it.

Can I change weights after the pipeline has run?

Yes, adjust the sliders in Settings and hit Save & Recompute. Only the affected presets are recomputed; the full pipeline doesn't re-run.

What's the difference between the graph and the embedding map?

The graph shows topological similarity: communities form from chains of strong connections. The embedding places every artist by their raw feature vector, independently of the graph. They capture different structure and only partially overlap, so both views are worth using.

Can I use this with data outside of Spotify?

It relies on Spotify audio features. If you can produce a normalized_tracks.json matching the expected schema, it will work with any source.
